Design Reference
AI Use Cases — Complete Reference
20 use cases derived from the 24-entity, 280-field data model. Every entity is covered. No data is collected but unanalysed. This document is the authoritative specification for what each insight skill reasons over, what it writes back, and its UX implications.
Architecture Note — Session 14 (2026-04-02)
Unified Skill Model: These 20 use cases are implemented as insight skills within capability-based agents, not as a separate “intelligence layer.” Each use case is a skill — a Claude API call with a specific prompt, MCP tool access, and AIInsight output schema — identical in implementation to the productivity skills (ITT extraction, response drafting, etc.) derived from the AI suitability assessment. The difference is trigger type (on-demand/scheduled vs human-invoked) and data direction (reads internal data only vs reads uploaded documents). See pwin_architect_plugin_architecture.md Section 7 for the full rationale.

Agent mapping: UC1-7, UC9, UC12 → Agent 3 (Strategy & Scoring). UC10-11 → Agent 4 (Commercial & Financial). UC13-14 → Agent 1 (Document Intelligence). UC15 → Agent 5 (Content & Response Drafting). UC8 → Agent 6 (Solution & Delivery). UC16-20 → Deferred (V2/V3, requires multi-bid data).
Key Rule
Claude writes only to AI-owned fields. The MCP server rejects any write to bid manager-owned fields at the API layer. AIInsight records are always append-only — never updated or deleted. Each analysis run creates a new record with a generated_at timestamp.
Framing
Use Cases 16–20 (Theme E) generate individual-level performance data. The ai_narrative field for these use cases must always be framed as coaching and resourcing intelligence, not performance evaluation. These insights are restricted to bid manager and bid director roles only.

Product Architecture — Unified Skill Model
All AI capabilities — whether extracting documents, drafting responses, or analysing timeline risk — are implemented as skills within six capability-based agents. The difference between skill types is configuration, not infrastructure.
1
Bid Execution App
Data & UI Layer
Data model: 24 entities, ~280 fields. Bid manager enters data through methodology activities. Single self-contained HTML file. localStorage persistence. The data source for all skills.
2
Six Agents — ~40 Skills
Productivity + Insight Skills
Six capability-based agents, each owning a library of skills. Productivity skills (human-invoked, produce deliverables) and insight skills (on-demand/scheduled, write AIInsight records) share the same implementation pattern: prompt + MCP tools + Claude API call.
3
MCP Server
Shared Infrastructure
Data access layer with filtered queries. Field-level permission enforcement (owner: bid_manager | ai). AIInsight append-only writes. Serves all agents and all skill types identically.
Unified Skill Model
The 20 insight skills in this document and the ~25 productivity skills from the AI suitability assessment are implemented identically. Both use the same MCP server, the same Claude API, and the same agent configurations. The distinction is product design (who triggers it, what it reads, what it produces), not engineering. See pwin_architect_plugin_architecture.md Section 7 for the full rationale.
Theme A — Schedule and Delivery Intelligence
Uses the Activity spine, dependency graph, and workstream structure.
01
UC1 — Preventative Timeline Analysis
Plugin Role 1 • Trigger: Scheduled weekly + on demand via /pwin:timeline-review
Data Consumed
  • Activity (plannedStart/End, actualStart/End, status, dependencies[], parallelisationType, effortDays, isCriticalPath, lastUpdated)
  • Workstream (timingHealth)
  • BidCalendar (publicHolidays, teamUnavailability)
What Claude Reasons Over
  • Gap in working days (not calendar days) between planned and forecast completion
  • Distinction between late-but-recoverable and past the point of recovery
  • Dependency cascade — compound slip across the full dependency chain
  • Rule violation detection — sequencing violations not just date violations
  • Predictive warning — on-time activities with risk characteristics (insufficient buffer, shared resource, unconfirmed third-party dependencies)
  • Critical path prioritisation — one day on critical path weighted as more serious than three days on a non-critical workstream
Written Back to Application
  • ActivityAIInsight: gap_working_days, rag_status, cascade_risk, downstream_impact[], ai_narrative, mitigation_options[], confidence, generated_at
Design Implication
RAG flag on Gantt row and activity detail panel. Mitigation options as selectable cards — accepting one triggers a StandupAction. Context window constraint: use filtered MCP queries only, never load all 79 activities at once.
02
UC2 — Effort Reforecasting and Resource Conflict Detection
Plugin Role 1 • Trigger: When actualStart/actualEnd diverges from plan on any activity
Data Consumed
  • Activity (effortDays, teamSize, parallelisationType, owner, plannedStart/End, actualStart/End)
  • TeamMember (name, workstreams[], individualDayRate, isContractor, startDate, endDate)
  • RateCard (roleName, standardDayRate)
  • BidCalendar (teamUnavailability)
What Claude Reasons Over
  • Reforecasts remaining effort as actual dates diverge from plan
  • Identifies concurrent specialist (S-type) activities assigned to the same person — cannot be parallelised as schedule assumes
  • Calculates compound impact: one activity slipping consumes resource needed for a downstream activity
  • Models implied capacity shortfall from overrun pattern
  • Calculates revised bid cost estimate
Written Back to Application
  • Revised forecast dates per activity (AI-owned field, separate from plannedEnd)
  • Resource conflict register: { person, activities[], overlap_days, severity, recommended_resolution }
  • Revised bid cost estimate to Engagement.healthStatus
Design Implication
Resource conflict register renders in Team Manager → Spend Forecast tab. Colour-coded by severity. Recommended resolution is one specific, actionable sentence per conflict.
03
UC3 — Intelligent Standup and Action Prioritisation
Plugin Role 1 • Trigger: Scheduled daily at 08:00 working days + on demand
Data Consumed
  • StandupAction (owner, dueDate, status, source, createdDate, completedDate, parentActivityCode)
  • Activity (status, blockedBy, dependencies[], plannedEnd, isCriticalPath)
  • GovernanceGate (status, sequence, entryCriteria[])
What Claude Reasons Over
  • Not a status report — a prioritised action list: which actions, if done today, unblock the most downstream work
  • Carry-forward actions deferred more than twice require an explicit bid manager decision (complete / cancel / escalate)
  • Gate proximity weighting: actions linked to a GovernanceGate within 5 days escalated regardless of activity criticality
Written Back to Application
  • Daily brief to BM workstream: max 5 items ranked by impact, each with named owner, specific ask, and due time
  • Carry-forward flags on deferred StandupAction records: deferred_count, escalation_recommended, recommended_decision
Design Implication
Brief renders on Workspace home screen as a prioritised list above the standup tracker. Bid manager can mark items done from the brief card without navigating to the tracker.
Theme B — Proposal Quality and Scoring Intelligence

04
UC4 — Compliance Coverage Intelligence
Plugin Role 1 • Trigger: On demand + at each review cycle gate
Data Consumed
  • ResponseSection (evaluationMaxScore, hurdleScore, evaluationCategory, evaluationCriteria, wordLimit)
  • ResponseItem (status, effortDays, complexityEstimate, qualityAnswersQuestion)
  • EvaluationFramework (qualityWeightPct, priceWeightPct, socialValueWeightPct)
  • ComplianceRequirement (classification, coverageStatus, complianceStatus)
What Claude Reasons Over
  • Weighted coverage risk: marks available × coverage status × days to submission — not binary covered/not covered
  • A 30% weighted question at draft_in_progress with 8 days to submission is categorically different from a 5% weighted question in the same state
  • Hurdle questions receive disproportionate effort regardless of mark value — below-hurdle is disqualification
  • Mandatory compliance gaps with complianceStatus: non_compliant are submission-level disqualification risks
Written Back to Application
  • Weighted coverage risk matrix per ResponseSection: marks at risk, days to submission, recommended action
  • Compliance gap register: non-compliant mandatory requirements flagged with disqualification risk label
  • Rendered as heat map overlay on Response Register
Design Implication
Heat map renders in Submissions module. Colour intensity reflects marks-at-risk severity, not binary coverage. Disqualification risks render as a persistent banner on the module header until resolved.
05
UC5 — Win Theme Integration Audit
Plugin Role 1 • Trigger: On demand + at storyboard approval (SAL-06 completion)
Data Consumed
  • WinTheme (reference, statement, rationale, status, linkedResponseItemIds[])
  • ResponseItem (qualityWinThemeAssessments[], status, owner)
  • ResponseSection (evaluationMaxScore, evaluationWeightPct, evaluationCategory)
  • ReviewScorecard (scores.win_themes per response per cycle)
What Claude Reasons Over
  • Win theme penetration across all responses: which themes are landed with substantiation vs. name only
  • High-weighted questions with no win theme — strategic gap, not drafting gap
  • Theme over-concentration in low-weighted sections where they generate few marks
  • Cross-references integration heatmap against evaluation criteria weighting
Written Back to Application
  • Win theme coverage score per ResponseSection (AI-owned field)
  • Portfolio-level theme saturation rating per WinTheme
  • Ranked list: highest impact sections where theme integration is weakest
Design Implication
Existing heatmap gains an AI coverage score column. Partial integration cells receive a recommended action tooltip. Ranked priority list renders below the heatmap as a coaching card.
06
UC6 — Marks Allocation Strategy Optimisation
Plugin Role 1 • Trigger: At ITT ingestion completion or storyboard stage
Data Consumed
  • ResponseSection (evaluationMaxScore, hurdleScore, evaluationCategory, wordLimit)
  • ResponseItem (effortDays, complexityEstimate, status)
  • EvaluationFramework (category weights)
  • ReviewScorecard (averageScore, predictedClientScore — current trajectory)
What Claude Reasons Over
  • Return on effort per response: marks available ÷ estimated effort days
  • Across 30-40 sections, trade-offs are invisible without AI reasoning across all simultaneously
  • Hurdle questions receive disproportionate effort regardless of return calculation
  • Recommendations are specific: ‘move 2 person-days from Q4.1 (5 marks, 8 days effort) to Q2.3 (25 marks, 4 days effort)’
Written Back to Application
  • Effort allocation efficiency score per ResponseItem (marks per effort-day)
  • Ranked reallocation table: over-invested sections, under-invested sections, hurdle exceptions
  • Recommended effort transfer: specific person-day quantities between named sections
Design Implication
Renders in Submissions module as an ‘Effort Efficiency’ panel. Actionable — accepting a recommendation updates ResponseItem.effortDays and triggers a StandupAction to the relevant owner.
07
UC7 — Review Quality Trajectory Analysis
Plugin Role 1 • Trigger: After each ReviewCycle closes
Data Consumed
  • ReviewScorecard (reviewer, scores — all 8 dimensions, averageScore, passesThreshold, predictedClientScore, predictedClientGradeBand, meetsHurdle)
  • ReviewAction (criterion, severity, status, owner, resolutionNotes)
  • ReviewCycle (type, scheduledDate, overallScore)
  • ResponseItem (owner, qualityScoreGap)
  • ClientScoringScheme (gradeBands[])
What Claude Reasons Over
  • Trajectory per response: pink → red → gold score movement on each of the 8 quality dimensions
  • Flatlined responses: pink to red improvement < 0.3 points indicates feedback is not landing
  • Critical gap: responses where internal quality score passes threshold but predicted client score does not
  • Grade band proximity: responses one band below ‘Excellent’ where specific evidence additions would close the gap
Written Back to Application
  • Trajectory assessment per ResponseItem: improving / plateauing / declining per dimension
  • Below-hurdle flags with specific gap-closing recommendation
  • Recovery priority list: responses where targeted intervention yields highest marks improvement
Design Implication
Trajectory renders in Reviews module on cycle detail view as a comparison chart (pink vs red vs gold per dimension). Below-hurdle flags are persistent alerts on the ResponseItem row in the production pipeline.
08
UC8 — Reviewer Calibration and Bias Detection
Plugin Role 1 • Trigger: 48 hours before each ReviewCycle • consolidate with UC17
Data Consumed
  • ReviewScorecard (reviewer, scores — all 8 dimensions, predictedClientScore, averageScore — all past cycles)
  • ReviewCycle (type, reviewers[])
  • ReviewAction (severity, criterion — attributed to reviewer)
What Claude Reasons Over
  • Systematic dimensional bias: reviewer consistently scoring below team average on specific dimensions regardless of response quality
  • Optimism/pessimism in predictedClientScore: consistently higher or lower than consensus
  • Severity calibration: disproportionately high critical action rates indicate miscalibrated standards
  • Predictive accuracy: whose past predicted scores most closely matched actual evaluator feedback
Written Back to Application
  • Reviewer calibration brief per cycle: individual known tendencies, recommended interpretation adjustments
  • Predictive accuracy ranking: updated post-award when actual scores are available
Design Implication
Calibration brief renders in Reviews module as a pre-cycle panel visible only to the bid manager. Not surfaced to reviewers. Interpretation aid for the bid manager when dissenting scores appear.

UC9 — Gate Readiness Pre-Assessment
Plugin Role 1 • Trigger: 5-7 working days before each GovernanceGate (scheduled)
Data Consumed
  • GovernanceGate (gateType, entryCriteria[], status, riskPremiumPct)
  • Activity (status, outputAssurance, owner — for activities in entry criteria)
  • ReviewCycle (status, overallScore)
  • ReviewAction (severity, status — critical open actions)
  • Stakeholder (stakeholderType, linkedGateIds[], isActive)
What Claude Reasons Over
  • Per entry criterion: current pass / fail / at-risk status with recovery assessment
  • Criteria failing with insufficient recovery time — gate deferral or scope narrowing decision required
  • Process integrity: activities showing outputAssurance: assured where reviewer and owner are the same person — conflict of interest
  • Governance gaps: gates with no linked approver or approver with isActive: false
Written Back to Application
  • Gate readiness report to GovernanceGate record: green (met), amber (at-risk + action), red (failing + escalation), process integrity concerns
  • StandupActions auto-generated for each recoverable amber criterion
Design Implication
Gate readiness report renders in Governance module as a pre-gate briefing panel, available 5 days before gate date. Exportable as a one-page PDF for the chair.

UC10 — Risk Register Pattern Detection and Pricing Validation
Plugin Role 1 • Trigger: At Gate 3 (commercial/risk review) + on demand
Data Consumed
  • RiskAssumption (registerType, probability, impact, status, mitigationOrValidation, impactIfWrong)
  • GovernanceGate (gateType: commercial_review, riskPremiumPct, decision)
  • Engagement (costToBid, winFeeValue, feeModel, healthStatus)
What Claude Reasons Over
  • Cross-references risk register against stated risk premium: HIGH probability / HIGH impact risks with status: open at Gate 3 indicate under-priced risk
  • Probability-weighted impact of all open items estimates defensible premium range
  • Assumption validation gaps: assumptions with high impactIfWrong that are unvalidated before Gate 3 are unacknowledged price risk
  • Delivery vs bid risk register divergence may represent unpriced delivery risk
Written Back to Application
  • Risk-to-premium consistency verdict: consistent / inconsistent / insufficient data
  • Defensible premium range: { min_pct, recommended_pct, max_pct } with rationale
  • Unvalidated assumption list requiring resolution before gate can be recommended as approve
Design Implication
Risk premium analysis renders in Governance module at the commercial review gate. Decision-support panel for the approving body — data-validated challenge, not a recommendation to override commercial judgment.

UC11 — Bid Cost and Spend Forecasting Intelligence
Plugin Role 1 • Trigger: Weekly + when team composition or activity effort changes
Data Consumed
  • TeamMember (individualDayRate, isContractor, workstreams[], startDate, endDate)
  • RateCard (roleName, standardDayRate)
  • Activity (effortDays, teamSize, owner, plannedStart/End, actualStart/End)
  • Engagement (costToBid, winFeeValue, feeModel, healthStatus, interventions[])
What Claude Reasons Over
  • Burn rate against costToBid: at current pace, will the bid investment authorisation be exceeded before submission?
  • Contractor premium risk: where contractor rates exceed the standard card on critical path activities, delay compounds the premium
  • P&L impact of scope expansions: models return impact before a commitment is made
  • Return on bid investment: win fee minus forecast cost at current trajectory
Written Back to Application
  • Budget burn rate forecast: current spend, projected total, variance from costToBid
  • Contractor premium analysis: named individuals, critical path overlap, estimated exposure
  • P&L scenario to Engagement.healthStatus and Engagement.interventions[]
Design Implication
Renders in Team Manager → Spend Forecast tab as an AI analysis panel below the existing breakdown. Scenario card allows bid manager to model a proposed change and see P&L impact before committing.

UC12 — Stakeholder Engagement and Gate Risk Assessment
Plugin Role 1 • Trigger: 10 days before each GovernanceGate + weekly scan
Data Consumed
  • Stakeholder (stakeholderType, linkedGateIds[], isActive)
  • GovernanceGate (gateType, sequence, status, approvingBody)
  • TeamMember (securityClearance, clearanceStatus, workstreams[])
  • Bid (securityClearanceTier, isRebid, incumbentName)
What Claude Reasons Over
  • Gate participation gaps: no linked approver, or only approver has isActive: false — will delay the gate
  • Clearance mismatches: Bid.securityClearanceTier is SC or DV but TeamMember records show clearanceStatus: in_progress
  • Sponsor disengagement: sponsor not linked to any upcoming gate past bid midpoint — PWIN risk
  • Rebid complacency: isRebid: true with no new stakeholder relationship-building activity recorded
Written Back to Application
  • Gate participation gap flags per GovernanceGate: missing approver, inactive approver, no reviewer linked
  • Clearance risk register: named individuals, affected workstreams, risk description
  • Sponsor engagement recommendation: StandupAction to BM workstream if disengagement detected
Design Implication
Gate participation gaps render in Governance module as a pre-gate warning banner. Clearance risks render in Team Manager → Directory as a flag on the individual row. Sponsor recommendation creates a StandupAction automatically.

UC14 — ITT Amendment Detection and Risk Flagging
Plugin Role 3 • Trigger: New ITTDocument added with source: clarification_pack or version increments
Data Consumed
  • ResponseSection (lastAmended, evaluationMaxScore, hurdleScore, evaluationCriteria, wordLimit, questionText)
  • ITTDocument (version, source, receivedDate, parsedStatus)
  • EvaluationFramework (category weights)

Amendment log renders in ITT Ingestion module as a version history timeline. Material changes trigger a persistent notification on the Workspace home screen until acknowledged.

UC19 — Workstream Lead Performance Consistency
Plugin Role 1 • Trigger: Weekly + when 3+ activities in a workstream show divergence
Data Consumed

Implementation Challenges
Challenge 1
Single-file vs Plugin Boundary
The bid execution HTML app is the data layer and UI. The AI use cases run in a separate plugin/service via MCP. The HTML app must expose its localStorage data to the MCP server — requires a bridge API or shared persistence layer.
Challenge 2
MCP Query Granularity
UC1 specifies “never load all 79 activities at once.” The MCP server needs filtered queries: per-workstream, per-gate, per-cycle, per-response-section. Each use case needs a defined query contract.
Challenge 3
Scheduled Triggers
UC1 (weekly), UC3 (daily), UC8 (48h before review), UC12 (10 days before gate). A browser-based app has no scheduler. V1 launches all use cases as on-demand only. Scheduled execution is V2.
Challenge 4
Context Window Management
Several use cases need portfolio-level reasoning (30-40 sections simultaneously). Feasible with current models but MCP query design must minimise payload while preserving reasoning completeness.
Challenge 5
Theme E Data Maturity
UC16-20 require multi-bid data for reliable signal. Build data collection infrastructure in V1, surface intelligence in V2/V3 when 3+ bids per individual available.
Note
UC8/UC17 Duplication
